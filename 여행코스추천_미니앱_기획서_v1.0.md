# 지역·예산 기반 AI 여행 코스 추천 미니앱 기획서

**버전**: v1.0
**작성일**: 2026-07-01
**대상 플랫폼**: 앱인토스 미니앱 (토스 인앱 웹뷰)
**출품 대상**: 앱인토스 7월 바이브코딩 챌린지 "미니앱과 함께하는 여름" (출품기간 7/1~7/29)

---

## 1. 문서 개요

### 1.1 목적
여름 휴가지를 정할 때 겪는 가장 번거로운 문제 — "정보는 많은데 내 예산과 취향에 맞게 걸러줄 사람이 없다" — 를 해결하는 미니앱을 정의한다. 단순 정보 나열형 앱이 아니라, **사용자의 소비 성향(숙박/식비/관광지 비중)을 파악해 같은 예산이라도 사람마다 다른 코스를 추천**하는 것이 핵심 차별점이다.

### 1.2 핵심 가치제안
| 기존 서비스 | 이 앱 |
|---|---|
| 네이버지도/카카오맵: 정보는 풍부하나 예산 반영 없음 | 예산 입력 시 자동으로 배분·추천 |
| 트리플/대한민국구석구석: 코스 추천은 하나, 성향 미반영 | 성향 테스트 기반 맞춤 코스 3개 |
| 일반 예산 계산기: 정산만, 코스 추천 없음 | 예산과 코스를 동시에 설계 |

### 1.3 타겟 유저
국내 근거리·당일치기~1박2일 여름 여행을 계획하는 개인 여행자. 여행 정보 서치에 드는 시간을 줄이고 싶은 사람.

---

## 2. 전체 유저 플로우

```
[1단계] 지역·여행기간·인원수·예산·이동수단 입력
        └ 교통비 포함 여부 선택
              ↓
[2단계] 5문항 성향 테스트 (숙박/식비/관광지 비중 산출)
        └ (백그라운드) TourAPI로 해당 지역 데이터 프리페치
              ↓
[3단계] AI 코스 생성 (Claude API)
        └ 성향 비율 + 순예산 + 여행기간 + 인원수 + 이동수단 제약 반영
              ↓
[4단계] 결과: 코스 카드 3개 (메인 1 + 서브 2) + 공유 버튼
```

## 3. 화면별 상세 명세

### 3.1 입력 화면

**화면 목적**: 지역, 여행 기간, 인원수, 예산, 이동수단, 교통비 포함 여부를 받아 이후 단계의 계산 기준을 확정한다.

**입력 필드**

| 필드명 | 입력 유형 | 옵션값 | 필수 |
|---|---|---|---|
| 지역 | 검색형 셀렉트 | 시/도 + 시군구 (TourAPI 지역코드 기준) | Y |
| 여행 기간 | 라디오 | 당일치기 / 1박2일 / 2박3일 이상 | Y |
| 인원수 | 숫자 입력 또는 스텝퍼 | 1인 / 2인 / 3인 이상 (직접 입력) | Y |
| 예산 | 숫자 입력 또는 구간 선택 | 자유 입력, 만원 단위, **총액 기준** (예: "대략 15만원, 전체 인원 합산") | Y |
| 이동수단 | 라디오 | 자차 / 대중교통 | Y |
| 교통비 포함 여부 | 라디오 | 포함 / 별도 | Y (이동수단 선택 후 노출) |

**검증 규칙**
- 예산 미입력 시 진행 불가, 최소 안내 문구 표시 ("대략적인 금액이면 충분해요, 전체 인원 합산 금액이에요")
- 지역 미선택 시 다음 단계 비활성화
- 여행 기간·인원수는 이후 단계(성향 테스트 계산, AI 코스 생성)의 필수 파라미터이므로 기본값 없이 명시적 선택 요구

**설계 원칙**
- 예산은 **총액 기준**으로 통일한다. 인원수는 예산 계산에 관여하지 않고, AI 코스 생성 시 "이 인원이 다니기 좋은 장소"(단체석, 인원제한 액티비티 등) 필터링 정보로만 사용한다 — 인당 환산·룸 분배 계산까지 들어가면 복잡도가 커지므로 MVP 범위에서 제외.

**백그라운드 동작**
- 지역이 확정되는 즉시 TourAPI 지역기반관광정보조회(areaBasedList) 호출 시작 → 2단계 테스트 진행 중 완료되도록 프리페치

---

### 3.2 성향 테스트 화면

**화면 목적**: 5문항을 통해 숙박(L)/식비(F)/관광지·체험(A) 3축의 예산 배분 비율을 산출한다.

**문항 및 배점**

| 문항 | 선택지 | 배점 |
|---|---|---|
| Q1. 숙소는 어떤 느낌이면 좋으세요? | A. 가성비 게스트하우스, 어차피 잠만 잘 건데 | F+1, A+1 |
| | B. 그래도 조식 나오는 편안한 숙소 | L+2 |
| Q2. 맛집 앞 웨이팅 2시간, 어떻게 하실래요? | A. 기다려서 먹는다, 이게 여행의 낙이지 | F+2 |
| | B. 포기하고 다른 데로, 시간 아까워 | A+2 |
| Q3. 여행지에서 하루 스케줄은? | A. 가능한 많이 돌아보기, 알차게 | A+2 |
| | B. 느긋하게 카페에서 시간 보내기 | L+1, F+1 |
| Q4. 여행 가서 절대 아끼고 싶지 않은 건? | A. 숙소 | L+2 |
| | B. 음식 | F+2 |
| | C. 체험/입장료/액티비티 | A+2 |
| Q5. 예상보다 돈이 좀 남았어요. 어디에 더 쓰고 싶으세요? | A. 숙소 업그레이드 | L+2 |
| | B. 맛집 하나 더 | F+2 |
| | C. 체험 추가 | A+2 |

**점수 계산 로직**
1. 5문항 답변 후 L, F, A 각 축의 총점 합산
2. **당일치기 선택 시**: L(숙박) 점수를 계산에서 제외하고, F/A 두 축만으로 비율 재계산 (문항 구성은 그대로 두되 계산 단계에서만 분기 처리 — 별도 문항 세트를 만들지 않아 개발 복잡도를 낮춤)
3. 전체 합계 대비 비율(%)로 환산: `축 비율 = 축 점수 / (해당 축들의 총합) × 100`
4. 보정: 특정 축이 15% 미만이면 15%로 상향, 초과분은 최고 점수 축에서 차감 (특정 항목이 0에 가까우면 코스가 부자연스러워지는 것 방지). 당일치기의 경우 이 보정은 F/A 두 축에만 적용

**결과 라벨링** (최고 점수 축 기준)
- L 최고 → **호캉스 우선형**
- F 최고 → **미식 우선형**
- A 최고 → **알뜰 다다익선형**

라벨은 화면에 재미 요소로 노출하고, 실제 %배분 수치는 코스 생성 단계에서 내부적으로 사용한다.

---

### 3.3 로딩(코스 생성) 화면

**화면 목적**: AI 코스 생성 진행을 자연스럽게 체감시키는 인터벌 화면. 백그라운드에서 실제 API 호출이 진행된다.

**처리 순서**
1. 순예산 계산 (4.1절 참고)
2. TourAPI 프리페치 결과 + 순예산 + 여행기간 + 인원수 + L/F/A 비율 + 이동수단 제약을 Claude API에 전달
3. 응답 파싱 → 코스 카드 데이터 구성

**로딩 연출**: 1.5~2초 내외, "코스를 짜는 중..." 류의 문구 (실제로는 API 응답 시간에 맞춰 유동적으로 조정)

---

### 3.4 결과 화면

**화면 목적**: 메인 성향 코스 1개와 다른 스타일의 서브 코스 2개를 카드 형태로 보여주고 공유를 유도한다.

**코스 카드 구성 요소**
| 요소 | 설명 |
|---|---|
| 성향 라벨 | 예: "미식 우선형 코스" |
| 예산 배분 요약 | "숙박 25% · 식비 42% · 관광지 33%" + 대략 금액 |
| 장소 리스트 | 순서, 장소명, 타입(숙박/식당/관광지), 대략 비용대 |
| 이동 동선 안내 | 이동수단 제약 반영 (대중교통 접근 용이 여부 텍스트 안내) |
| 공유 버튼 | 앱인토스 공유 브릿지 SDK 연동 |

**메인/서브 구성 로직**: 테스트로 산출된 성향을 메인 코스로, 나머지 2개 성향(L/F/A 중 상위가 아닌 두 축을 각각 최고점으로 가정한 코스)을 서브 코스로 동일 API 호출에서 함께 생성 요청한다.

## 4. 핵심 로직 명세

### 4.1 순예산 계산 (교통비 처리)

이동수단과 무관하게 "교통비를 이번 예산에 포함할지"는 독립적인 선택지로 처리한다.

| 조건 | 처리 |
|---|---|
| 교통비 포함 선택 | 총예산에서 교통비 어림값 차감 후 잔액을 L/F/A에 배분. 어림값: 대중교통 총예산의 10~15%, 자차 5~10% (지역 원거리 여부에 따라 가변) |
| 교통비 별도 선택 | 입력 예산 100%를 그대로 L/F/A에 배분, 교통비는 계산에서 제외 |

### 4.2 이동수단에 따른 코스 추천 범위 제약
- 자차: 대중교통 접근성 제약 없이 외곽·근교 명소 포함 가능
- 대중교통: 역세권/버스 접근이 용이한 장소 위주로 필터링

이 규칙은 Claude API 프롬프트에 조건문으로 명시한다.

### 4.3 여행 기간·인원수 반영 방식
- **여행 기간(당일/1박2일/2박3일 이상)**: 숙박비를 정밀하게 박수만큼 곱해 계산하지 않고, "총 N박"이라는 정보를 AI 프롬프트에 그대로 전달해 그 범위 안에서 배분하도록 위임한다. 당일치기는 4.1의 계산 로직에 따라 L축이 제외된 상태로 전달된다.
- **인원수**: 예산 계산에는 관여하지 않는다(예산은 총액 기준). AI 프롬프트에 "인원 N명"으로 전달해 단체석 여부, 인원제한 액티비티 필터링 등 장소 적합성 판단에만 활용한다.

### 4.4 AI 코스 생성 프롬프트 설계 방향
**입력으로 전달할 데이터**
- TourAPI에서 가져온 해당 지역의 관광지/숙박/음식점 목록 (명칭, 타입, 주소, 대략 요금 정보 포함 시 첨부)
- 순예산 (교통비 차감 후, 총액 기준)
- 여행 기간 (당일/N박)
- 인원수
- L/F/A 배분 비율 3세트 (메인 성향 + 서브 성향 2개) — 당일치기는 F/A 2축만 전달
- 이동수단 제약 조건

**출력 형식**: 코스 3개를 JSON으로 반환하도록 구조화된 프롬프트 사용 (파싱 안정성 확보). 각 코스는 장소 리스트 + 대략 예산 배분 값을 포함.

**기대치 관리**: "정확한 금액 일치"가 아니라 "대략적인 예산대 안내"를 목표로 한다. TourAPI의 요금 데이터가 자유 텍스트로 정리가 안 된 경우가 많아, 정밀한 합산 대신 범위형 표현("이 코스면 대략 15만원선")을 사용한다.

## 5. 외부 연동 시스템

| 시스템 | 역할 | 연동 방식 | 비고 |
|---|---|---|---|
| 한국관광공사 TourAPI 4.0 | 관광지/숙박/음식점 데이터 | REST API, 공공데이터포털 인증키 | 개발 단계 자동 승인, 관광지·숙박·음식점 15종 데이터 통합 제공 |
| Anthropic Claude API | 예산·성향 기반 코스 생성 | REST API | ato 프로젝트에서 사용한 방식과 동일 구조 재사용 |
| 앱인토스 SDK | 결과 공유하기 브릿지 | 앱인토스 개발자센터 SDK | 필요 시 '사용자 정보' SDK로 간편 개인화 가능 |

## 8. 챌린지 심사 대응 전략

| 심사 기준 | 대응 근거 |
|---|---|
| 주제 적합성 (여름 여행·휴가 계획) | 지역·예산 입력 기반 코스 추천은 주제 가이드 1번 항목에 직접 부합 |
| 핵심 기능 존재 (리워드성 앱 배제 기준) | 성향 테스트는 재미 요소이며, 핵심 기능은 예산 기반 실제 코스 생성 — 테스트 단독이 아닌 실용 기능이 중심 |
| 1차 심사 (8/1~8/26 트래픽 지표) | 결과 코스 카드의 공유 버튼이 자연스러운 유입 경로 역할 |
| 최종 심사 (완성도·차별성) | 정보 나열이 아닌 "예산 배분 로직" 자체가 기존 지도/여행 앱과의 차별점 |

## 9. 리스크 및 대응

| 리스크 | 내용 | 대응 |
|---|---|---|

## 11. Claude Design 작업용 화면 설계 가이드

이 섹션은 Claude Design으로 시안을 제작할 때 그대로 참고 자료로 활용하기 위해 정리한다. 로직·데이터는 위 섹션(3~4)을 따르되, 여기서는 레이아웃·컴포넌트·톤앤매너만 다룬다.

### 11.1 전체 톤앤매너
- 여름 시즌 캠페인 느낌: 밝고 시원한 톤 (코랄·틸 계열 추천), 캐주얼한 카드형 UI
- 정보 밀도가 높은 화면(결과 화면)이 있으므로 여백과 카드 구분을 명확히 해 답답해 보이지 않게 구성
- 참고: 초기 기획 단계에서 나온 "영수증 테스트" 컨셉의 바코드·영수증 질감 UI를 결과 화면 카드 디자인에 재활용 가능 (톤은 유지하되 실용 정보가 많아지므로 레이아웃은 더 단순화 필요)

### 11.2 화면 1 — 입력 화면
- **레이아웃**: 세로 스크롤 폼. 필드 순서 = 지역 → 여행 기간 → 인원수 → 예산 → 이동수단 → 교통비 포함 여부
- **컴포넌트**:
  - 지역: 검색형 셀렉트(자동완성)
  - 여행 기간 / 이동수단 / 교통비 포함 여부: 칩(chip) 형태 라디오 버튼
  - 인원수: 스텝퍼(+/-)
  - 예산: 숫자 입력, 원화 단위 자동 포맷팅, 입력창 하단에 "전체 인원 합산 금액이에요" 보조 텍스트
- **상태**: 필수 필드 미충족 시 하단 CTA 비활성화
- **CTA 문구**: "성향 테스트 시작하기"

### 11.3 화면 2 — 성향 테스트
- **레이아웃**: 문항 1개씩 풀스크린 카드, 상단에 진행률 표시(예: "2/5")
- **컴포넌트**: 2~3지선다 버튼형 선택지 (세로 나열), 상단 프로그레스 바
- **상태**: 문항 전환 시 가벼운 전환 애니메이션(페이드 또는 슬라이드)
- **당일치기 선택자 대응**: 숙박 관련 문항(Q1, Q4-A, Q5-A)도 화면상 노출은 동일 — 계산 단계에서만 제외되므로 디자인 시안에서 별도 분기 화면은 불필요

### 11.4 화면 3 — 로딩(코스 생성)
- **레이아웃**: 중앙 정렬, 애니메이션 + 짧은 문구
- **컴포넌트**: 로딩 인디케이터 (여행가방/지도/나침반 등 여행 테마 일러스트 활용 권장)
- **문구 예시**: "코스를 짜는 중..." / "예산에 맞춰 정리하고 있어요"

### 11.5 화면 4 — 결과 화면
- **레이아웃**: 메인 코스 카드(상단, 크게) + 서브 코스 카드 2개(하단, 스와이프 가능한 캐러셀 또는 탭 전환)
- **컴포넌트**:
  - 성향 라벨 배지 (예: "미식 우선형")
  - 예산 배분 시각화 (숙박/식비/관광지 비율 — 막대 또는 도넛 형태)
  - 장소 리스트 (순서 번호 + 아이콘 + 장소명 + 대략 비용대)
  - 이동 동선 보조 텍스트 (대중교통 접근 용이 여부)
  - 공유 버튼 (화면 하단 고정)
- **상태**: 메인 코스 기본 노출 → 서브 코스는 탭/스와이프로 전환

### 11.6 Claude Design 작업 시 참고 순서
1. 11.1~11.5 내용을 Claude Design에 그대로 전달해 화면 4종 시안 제작
2. 시안 확정 후 3~4절의 로직/데이터 명세를 기준으로 실제 프로그래밍(React+Vite) 진행
3. 시안 단계에서 확정된 컴포넌트 구조는 이 문서에 다시 업데이트 권장 (코딩 단계에서 재작업 방지)
