## 2026-07-21 — 결과 화면이 "임시 코스 → 진짜 코스"로 뒤늦게 바뀌던 문제 수정

사용자 피드백: "처음 결과가 이상하다 했는데 잠시 후 바뀌고, 조금 더 있다가 또 최종으로
바뀐다 — 추가 버튼도 처음엔 안 보인다. 시간 되면 무조건 결과를 보여주는 것 같다." 정확한
진단이었음.

- **원인**: AI 응답이 오면 그때 결과 화면으로 넘어가는 로직 자체는 이미 있었지만
  (`App.jsx`의 `finishLoading()`은 AI fetch의 `.then`/`.catch` 안에서만 호출됨), 그 위에
  로딩 화면 강제 탈출용 안전장치가 13초로 걸려 있었음. 서버 쪽 AI 호출 예산은
  `WALL_CLOCK_MS=12000`(aiPlan.js)인데 클라이언트 타임아웃은 13초라 여유가 1초뿐이었음 —
  실제 네트워크 왕복/처리 오버헤드를 더하면 서버가 제 예산 안에 응답을 만들어도 클라이언트
  타임아웃이 먼저 발동하는 경우가 흔했고, 그러면 규칙 기반 임시 코스를 먼저 보여준 뒤
  뒤늦게 도착한 AI 결과로 화면이 다시 덮어써지는 게 사용자 눈에 "결과가 바뀐다"로 보였음.
  추가 버튼(`+ 추가`)이 처음엔 안 보인 것도 같은 이유 — 임시(규칙 기반) 코스가 후보를 거의
  다 써버려서 추가할 게 없다가, AI 코스로 바뀌면서 남는 후보가 생겨 버튼이 나타난 것.
- **수정**: 안전장치 타임아웃을 13초→25초로 늘림. 서버 예산(12초)보다 충분히 여유를 둬서,
  "정상적으로 좀 오래 걸리는 경우"에는 절대 발동하지 않고 "응답이 아예 안 오는" 진짜
  예외적인 경우에만 걸리게 함 — 유저가 명시적으로 "기다리는 시간이 더 걸려도 되니 최종
  결과 한 번에 보고 싶다"고 요청한 방향과 일치.
- 검증: Playwright로 AI 응답이 실제로 ~12초 걸리는 경우까지 재현 → 응답 도착 6ms 뒤 바로
  최종 화면 표시, 표시 후 4초 더 지켜봐도 내용 변화 없음(`CHANGED AFTER DISPLAY: false`)
  확인. 빌드/`npm test`(23/23) 통과.

## 2026-07-20 — 실시간 장소 조회 실패 시 빈 코스 화면으로 떨어지던 문제 수정

로컬 PC 미리보기에서 백엔드가 일시적으로 꺼져 있었을 때, 코스 화면이 완전히 빈 목록(0원,

서버가 idle 상태에서 꺼지는 이 환경 특유의 문제였지만("Render 배포본엔 영향 없음"), 사용자가
"실제 서비스에서 이런 상황이 재현되면 안 된다"고 요청 — 원인을 구조적으로 점검.

- **발견한 진짜 버그(코드 수정)**: 실시간 데이터를 못 가져오면 "샘플 데이터"로 대체하는
  폴백 로직과 그걸 알려주는 `DataSourceBadge`("샘플 데이터")까지 이미 다 만들어져 있었는데,
  `CoursesScreen`의 장소 목록 필터(App.jsx)가 **소스와 무관하게 무조건 카카오 인증된 장소만**
  통과시키고 있어서, 샘플 데이터는 애초에 카카오 인증이 없으니 전부 걸러져 빈 화면이 되는
  버그였음 — 폴백 장치가 이미 있었는데 다른 코드가 그걸 무력화하고 있던 셈. 실시간
  (`course.source === 'tourApi'`) 모드에서만 카카오 인증 필터를 적용하도록 수정.
  Playwright로 `/api/tour-places` 호출 자체를 강제로 실패시켜 재현 → 수정 후 "샘플 데이터"
  배지 + 실제 예산 계산까지 정상 표시되는 것 확인.
- **카카오 조회 안정성 보강(`server/kakaoLocal.js`)**: 개별 장소의 카카오 인증 요청이
  실패(429/5xx/네트워크 오류)해도 재시도 없이 바로 포기하고 있었고, 실패해도 서버 로그에

  남기지 않게 배치 단위로 집계).

  걸리면 유사한 지연을 겪을 수 있음. 유료 플랜 전환 또는 주기적 warm-up ping 중 택1 필요 —

## 2026-07-14 — 하단 액션 버튼: absolute 오버레이 → flex 인라인 구조로 근본 수정

바로 이전 수정(`relative` 추가)을 Render에 배포해 실기기 사파리로 재검증했는데도, "인원수" 스텝퍼가

- **`src/App.jsx` (`BottomBar`)**: `absolute inset-x-0 bottom-0` 제거 → `relative shrink-0`로 변경.

  컨테이너의 `h-[100dvh]`와 정확히 일치하는 순수 flexbox 배치라 겹침이 구조적으로 불가능함(dvh/
  안전영역/WebKit 리사이즈 타이밍과 무관).
- **`InputScreen`/`PersonalityScreen`/`CoursesScreen`/`SavedCourseDetailScreen`**: BottomBar 높이를
  추정해 예약하던 `pb-[calc(Npx+env(safe-area-inset-bottom))]`를 전부 제거하고 단순 `pb-4`로 교체
  — 더 이상 BottomBar 실제 높이를 수동으로 맞출 필요 없음(CoursesScreen의 "바꾼 곳 있음" 주황

- 검증: 코스 화면에서 "빼기"를 눌러 BottomBar가 185px→233.5px로 실제로 늘어나는 상황까지 만들어

- **Kakao Local**: ✅ 정상. 실제 도로명주소·전화번호 보강.
- **OpenAI**: ✅ 정상. AI 예산 진단 문구 실시간 생성.

### 4. 안정성 보강
- **`server/tourApi.js`**: data.go.kr 5xx/네트워크 오류 시 백오프로 **최대 2회 자동 재시도**(`fetchWithRetry`) 추가. 최종 실패 시 서버 로그 기록.
- **`src/App.jsx`**: 결과 카드에 **데이터 출처 배지** 추가 — 실시간(TourAPI) 반영 시 "실시간 장소 반영", 실패 시 "샘플 데이터" 표시.

### 5. 후속 처리 (같은 날)
- **당일치기 호캉스 코스 제거** (`src/lib/courses.js`, `src/App.jsx`): 당일치기는 숙박 축(L=호캉스)을 코스에서 제외해 F/A 2개만 생성. 세그먼트 탭도 코스 개수에 맞춰 동적 컬럼으로.
- **문서 AI 엔진 표기 정리** (기획서 2부·`design_handoff/README.md`·`src/lib/courses.js` 주석): 런타임 "Claude API" 언급을 실제 구현(**OpenAI gpt-4.1-mini**)에 맞춰 수정. **AI 엔진은 OpenAI 유지(교체 아님).** 단, §11 "Claude Design"은 시안 제작에 쓴 도구 언급이라 사실이므로 유지.

### 6. 실제 지도 · 전국 지역 선택
- **Kakao Maps JS 실제 지도**(`src/App.jsx`): 결과 화면 placeholder → 실제 카카오 지도(TourAPI 좌표로 번호 마커+fitBounds). `VITE_KAKAO_MAP_KEY`(도메인 등록 공개키), 키/좌표 없으면 placeholder fallback.
- **전국 지역 선택**: 4개 하드코딩 → 전국 시/도 17 + 시/군/구 234개.
  - `scripts/generate-regions.mjs`로 TourAPI에서 코드 트리 생성 → `src/data/regions.json` (시/도명 정규화: 경기도→경기 등).
  - `server/tourApi.js`: 하드코딩 REGION_CODES → regions.json 기반 지역명→코드 매핑.
  - `src/App.jsx`: 단일 select → **바텀시트 드릴다운 선택기**(시/도→시/군/구 + 전국 검색). region 문자열 형식 유지로 지도·공유 호환.
  - 검증: 경기 평택/서울 강남/충남 태안 실데이터+지도+AI 확인.

### 참고
- 실제 API가 반영된 화면은 `npm run build && npm run serve`(포트 5175)로 확인. vite dev(5174)는 `/api/*`를 프록시하지 않아 항상 샘플 데이터.
- 서버 코드(`server/*.js`) 수정 시 실행 중인 serve 프로세스는 재시작해야 반영됨(지역 매핑 등).
